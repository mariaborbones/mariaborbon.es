## Architecture

- `index.html` ‚Äî the hero is a GSAP ScrollTrigger horizontal "walking scene" (`.scene` ‚Üí `#world`, pinned in place) telling Mar√≠a's day as 7 time-stamped zones (`zones` array in `main.js`, 06:00 gimnasio ‚Üí 06:30 casa ‚Üí 07:00 camino al trabajo ‚Üí 07:30 Verti ‚Üí 18:00 colegio ‚Üí 18:15 s√∫per ‚Üí 18:45 en casa). `.character-group` holds `.mum-jump > .figure.mum` (always present, walking in a fixed screen position while `.zone` buildings scroll past underneath), `.kids-group` (boy + girl + stroller), and `.family-car` (shown only during the 07:00 commute zone ‚Äî `DRIVING_ZONE` ‚Äî while `.mum` is faded out, since she's driving not walking). Four zones (gimnasio, Verti, colegio, s√∫per) each have a static Mario-style `.question-block` positioned near that zone's exit ‚Äî see below. The Verti zone (`.verti-office`, `.sign-verti`, `.verti-logo`) recreates Verti Seguros' real orange-to-red gradient wordmark in CSS (`background-clip: text`) rather than embedding their logo asset. After the pin, a plain `.evening` cards section covers the three remaining moments (19:30/20:00/20:30) that don't fit the walk-through-locations metaphor, then normal vertical `<section>`s (`#work`, `#contact`) hold Proyectos/Contacto.
- A fixed `.stats-hud` (top-right, "Mi d√≠a") shows four scroll-driven bars with numeric labels ‚Äî üîã battery, ‚ù§Ô∏è amor, üí™ salud, üßÝ conocimiento ‚Äî that dramatize the day: battery drains at every zone transition while amor/salud/conocimiento stay flat until earned. They're earned via **static reward boxes**, not mum moving: `rewardBoxes` in `main.js` gives each of the 4 `.question-block`s a fixed world-position (`zoneIndex * ZONE_WIDTH + BOX_CENTER_OFFSET`, matching where the CSS actually places `.question-block` inside `.zone`), and `mumCenterX` is measured once (mum's screen position never changes ‚Äî she's the one standing still while the world scrolls). Every frame, `checkRewardBoxes()` compares each box's current screen position (world position minus scroll distance) against `mumCenterX`; the instant a box reaches her, it fires once (`box.hit` flag ‚Äî Mario blocks don't come back), applies that box's stat jump, and plays `triggerBonk()`: `.mum-jump` (a wrapper kept separate from `.mum` itself so this doesn't fight the walk-cycle's own bob tween) hops up into the block, the block squashes and turns "used" (`.hit` class), and a matching emoji (`.stat-popup`) floats up from the block and fades. The colegio box additionally reveals `.kids-group` ‚Äî but that reveal is checked continuously (`checkKidsVisibility()`, reversible if you scroll back), separately from the one-shot bonk, since kids appearing/disappearing should track her actual walking position, not stay stuck on after a single trigger. The HUD fades out once `#work` (Proyectos) comes into view, since the "day" narrative is over by then.
- `assets/css/style.css` ‚Äî all visual design: the zone/building illustrations and the car (pure CSS shapes, no images), the character rig (`.figure .head/.body/.arm/.leg`, sized per `.girl`/`.boy`/`.mum`), the stroller, the sky's single dawn‚Üíday‚Üídusk gradient spanning the whole world width, the stats HUD, the question-block/popup (`.question-block`'s `left`/width in CSS must stay in sync with `BOX_CENTER_OFFSET`/`ZONE_WIDTH` in `main.js` ‚Äî they're not derived from each other), and the plain-section styles below the scene.
- `assets/js/main.js` ‚Äî GSAP setup: registers ScrollTrigger, pins `.scene` for a scroll distance equal to `world.scrollWidth - viewport width`, and on every scroll update drives (a) the world/sky/clouds/ground parallax via `backgroundPosition`/`x`, (b) a scroll-distance-driven walk cycle (leg/arm `rotate` via `Math.sin`, per-figure phase offset, stroller/car wheel spin), (c) the zone caption crossfade + progress bar, (d) `updateCharacters()`, which crossfades mum/car opacity based on which zone index is active, (e) `checkKidsVisibility()`/`checkRewardBoxes()` (see above), and (f) `updateStats(zones[zoneIndex].stats)` on every zone change, which keeps battery accurate even in zones without a reward box. A separate `setupEveningStats()` wires plain (non-pinned) `ScrollTrigger.create()` triggers on the `.evening-card`s so the HUD keeps updating past the pinned scene. GSAP + ScrollTrigger are loaded from the jsdelivr CDN (`<script>` tags in `index.html`), keeping the "no build step" approach ‚Äî don't introduce a bundler/npm install to add JS dependencies here.
- `render.yaml` ‚Äî deploys the site as a Render static site (`staticPublishPath: .`, no build command). Pushing to the connected branch auto-deploys.

Keep this structure flat: add new pages as sibling `.html` files, avoid introducing a build step, and when tweaking the walking scene's CSS-shape illustrations, verify visually (e.g. Playwright screenshots at several scroll fractions) rather than trusting the CSS by inspection alone ‚Äî the geometry is fiddly (nested `position: absolute` coordinate frames are a recurring source of bugs here) and looks nothing like the code until rendered.
