# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

María Borbonés's personal website — a plain HTML/CSS/JS static site with no build step and no dependencies.

## Commands

- Preview locally: open `index.html` directly in a browser, or run `npx serve .` for a local server.
- No install, build, lint, or test commands — there is no package.json or build tooling.

## Architecture

- `index.html` — the hero is a GSAP ScrollTrigger horizontal "walking scene" (`.scene` → `#world`, pinned in place) telling María's day as 7 time-stamped zones (`zones` array in `main.js`, 06:00 gimnasio → 06:30 casa → 07:00 camino al trabajo → 07:30 Verti → 18:00 colegio → 18:15 súper → 18:45 en casa). `.character-group` holds `.mum-jump > .figure.mum` (always present, walking in a fixed screen position while `.zone` buildings scroll past underneath), `.kids-group` (boy + girl + stroller, hidden until she's walked past the school building near the end of the colegio zone — a continuous `KIDS_JOIN_PROGRESS` threshold in `main.js`, not a zone boundary, since she walks to school alone and it should read as picking the kids up there rather than snapping in at the next zone), and `.family-car` (shown only during the 07:00 commute zone — `DRIVING_ZONE` — while `.mum` is faded out, since she's driving not walking). The Verti zone (`.verti-office`, `.sign-verti`, `.verti-logo`) recreates Verti Seguros' real orange-to-red gradient wordmark in CSS (`background-clip: text`) rather than embedding their logo asset. After the pin, a plain `.evening` cards section covers the three remaining moments (19:30/20:00/20:30) that don't fit the walk-through-locations metaphor, then normal vertical `<section>`s (`#work`, `#contact`) hold Proyectos/Contacto.
- A fixed `.stats-hud` (top-right, "Mi día") shows four scroll-driven bars with numeric labels — 🔋 battery, ❤️ amor, 💪 salud, 🧠 conocimiento — that dramatize the day: battery drains steadily from 100% to ~6% while amor/salud/conocimiento climb at the moments that earn them (gym → salud, Verti → conocimiento, kids rejoining → amor climbing through the evening). Every time one of the three growth stats jumps, `.mum-jump` (a wrapper around `.figure.mum`, kept separate from the figure itself so this doesn't fight the walk-cycle's own bob tween on `.mum`) plays a Mario-style `triggerBonk()` timeline: mum hops up into a `.question-block` above her head, the block squashes, and a matching emoji (`.stat-popup`) floats up and fades. Values live per-zone in `zones[].stats` and per-evening-card in `eveningStats` in `main.js`; the HUD fades out once `#work` (Proyectos) comes into view, since the "day" narrative is over by then.
- `assets/css/style.css` — all visual design: the zone/building illustrations and the car (pure CSS shapes, no images), the character rig (`.figure .head/.body/.arm/.leg`, sized per `.girl`/`.boy`/`.mum`), the stroller, the sky's single dawn→day→dusk gradient spanning the whole world width, the stats HUD, the question-block/popup, and the plain-section styles below the scene.
- `assets/js/main.js` — GSAP setup: registers ScrollTrigger, pins `.scene` for a scroll distance equal to `world.scrollWidth - viewport width`, and on every scroll update drives (a) the world/sky/clouds/ground parallax via `backgroundPosition`/`x`, (b) a scroll-distance-driven walk cycle (leg/arm `rotate` via `Math.sin`, per-figure phase offset, stroller/car wheel spin), (c) the zone caption crossfade + progress bar, (d) `updateCharacters()`, which crossfades mum/car opacity based on which zone index is active (kids-group opacity is handled separately by the continuous `KIDS_JOIN_PROGRESS` check, not this function), and (e) `updateStats()`, which tweens the HUD bars/labels and calls `triggerBonk()` whenever a stat increases meaningfully vs. `lastStats`. A separate `setupEveningStats()` wires plain (non-pinned) `ScrollTrigger.create()` triggers on the `.evening-card`s so the HUD (and bonks) keep working past the pinned scene. GSAP + ScrollTrigger are loaded from the jsdelivr CDN (`<script>` tags in `index.html`), keeping the "no build step" approach — don't introduce a bundler/npm install to add JS dependencies here.
- `render.yaml` — deploys the site as a Render static site (`staticPublishPath: .`, no build command). Pushing to the connected branch auto-deploys.

Keep this structure flat: add new pages as sibling `.html` files, avoid introducing a build step, and when tweaking the walking scene's CSS-shape illustrations, verify visually (e.g. Playwright screenshots at several scroll fractions) rather than trusting the CSS by inspection alone — the geometry is fiddly (nested `position: absolute` coordinate frames are a recurring source of bugs here) and looks nothing like the code until rendered.
